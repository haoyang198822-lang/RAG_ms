# RAG-Challenge-2 本地改写版知识整理

> 说明：本文档基于当前 `RAG-cy` 目录内代码整理，重点总结相对冠军项目 `IlyaRice/RAG-Challenge-2` 的本地改写方向、实现方式和运行链路。当前仓库里已经能明确看到的改写，主要集中在：
>
> 1. PDF 解析链路从原始 Docling 流程，扩展为支持 **MinerU** 解析中文财报/报告
> 2. 检索、问答、重排、Embedding、LLM 调用，逐步切换为 **国内模型/国内 API**
> 3. 面向中文企业年报场景，对公司名抽取、页码引用、表格处理、数据目录进行了本地化调整

---

## 一、项目整体目标

这个项目仍然保留了冠军方案的核心思想：

- 先解析 PDF 报告，得到结构化内容
- 再进行分页、分块、建库
- 问题到来后，先做公司名识别，再检索该公司文档中的相关上下文
- 最后交给大模型生成结构化答案，并带上页码引用

但是本地版本的重点已经明显变成了“**中文企业知识库**”场景，尤其是：

- 报告语言从英文年报转向中文财报、中文证券研报
- 解析引擎从单一 Docling 流程，扩展到 MinerU
- 模型调用从 OpenAI / Gemini / IBM 为主，切换到 Qwen / DashScope 等国内接口

---

## 二、和冠军项目相比，本地版的核心改写点

### 1）文档解析链路改成支持 MinerU

原冠军项目的 README 里强调的是：

- 自定义 PDF parsing
- 使用 Docling 解析报告
- 生成结构化 JSON，再做后续规整

而本地版代码里新增了 `src/pdf_mineru.py`，说明已经引入了 MinerU 作为新的解析入口。

#### 新增文件
- `src/pdf_mineru.py`

#### 关键功能
- 通过 MinerU API 创建解析任务
- 轮询任务状态
- 下载解析结果 zip
- 自动解压
- 将 `full.md` 移动到本地 markdown 目录

#### 实现方式

MinerU 的流程是：

1. 将 PDF 文件名拼成云端地址
2. 调用 MinerU 的 `extract/task` 接口生成任务
3. 轮询任务状态直到 `done`
4. 下载 `full_zip_url`
5. 解压得到 `full.md`
6. 在 `pipeline.py` 中把它搬运到本地统一目录

对应实现见：
- `src/pdf_mineru.py`
- `src/pipeline.py` 的 `export_reports_to_markdown()`

#### 这意味着什么

原冠军项目更偏向“结构化 PDF 解析 + 原始 JSON 后处理”；
而本地版已经支持“**MinerU 生成 Markdown**”作为新的中间表示。

这对中文材料非常有帮助，因为中文年报、研报里常见：

- 密集表格
- 复杂页眉页脚
- 图表说明
- 中英文混排

MinerU 的 Markdown 输出更适合作为后续分块、检索、问答的统一文本输入。

### 2）解析后处理逻辑做了本地化改写

#### 文件
- `src/pdf_parsing.py`
- `src/parsed_reports_merging.py`
- `src/text_splitter.py`

#### 主要变化

本地代码仍保留 Docling 风格的结构化输出处理，但做了几项针对本地数据的调整：

##### a. 支持中文公司名元数据补全

`PDFParser._parse_csv_metadata()` 中，从 `subset.csv` 读取公司信息时，不再只依赖单一字段，而是兼容：

- `company_name`
- `name`

这说明本地数据集的字段名和冠军项目原始数据集不完全一致。

##### b. 自动补齐页码顺序

`PDFParser._normalize_page_sequence()` 会把缺失的页号补成空页，避免中间页缺失导致后续页码对齐混乱。

这在中文报告场景里很实用，因为 MinerU / Docling 在某些扫描页、空白页、图像页上可能会出现页序不连续。

##### c. 报告内容规整更偏“中文可读文本”

`PageTextPreparation` 会把解析后的块进一步重排成页文本，并处理：

- 表格
- 列表
- 脚注
- 页眉页脚
- 特殊 glyph / slash 命令

这部分逻辑有利于把 PDF 解析结果变成更适合中文检索的 Markdown / Text。

##### d. 表格序列化支持切换

`parsed_reports_merging.py` 里支持：

- 直接保留 Markdown 表格
- 或者把表格再序列化成描述性文本

这对中文财报特别重要，因为不少问题答案不在表头，而在表格列之间的语义关系里。

### 3）模型调用切换到国内 API 和模型

这是本地版最明显的改写之一。

#### 文件
- `src/api_requests.py`
- `src/retrieval.py`
- `src/reranking.py`
- `src/pipeline.py`
- `src/questions_processing.py`

#### 原项目的模型特点
冠军项目原始说明里，主要依赖：
- OpenAI
- Gemini
- IBM Watson / IBM Foundation Models

#### 本地版的变化
本地版已经加入并默认倾向使用：
- `dashscope`
- `qwen-turbo-latest`
- `text-embedding-v1`

也就是说，核心链路已经向国产模型迁移：

- 问答模型：Qwen
- 嵌入模型：DashScope Embedding
- 重排模型：Qwen

#### 具体实现

##### a. 问答统一入口 `APIProcessor`

在 `api_requests.py` 中，`APIProcessor` 支持多个 provider：

- `openai`
- `ibm`
- `gemini`
- `dashscope`

其中本地版默认是：
- `provider="dashscope"`

这意味着问答主链路默认走 DashScope。

##### b. DashScope 生成器 `BaseDashscopeProcessor`

该类使用：
- `dashscope.Generation.call()`
- `qwen-turbo-latest`

并在返回后尽量解析 JSON；如果不是 JSON，则降级返回：
- `final_answer`
- `step_by_step_analysis`
- `reasoning_summary`
- `relevant_pages`

这对中文结构化问答很实用。

##### c. 向量检索 embedding 改为 DashScope

`src/retrieval.py` 里的 `VectorRetriever` 增加了：
- `embedding_provider="dashscope"`

其 `_get_embedding()` 会调用：
- `dashscope.TextEmbedding.call(model="text-embedding-v1")`

说明向量库构建与查询已经可切换为国内嵌入模型。

##### d. LLM 重排也支持 DashScope

`src/reranking.py` 中 `LLMReranker` 默认 provider 也是 `dashscope`，使用：
- `qwen-turbo`

进行单块或多块重排。

## 三、改写后的完整数据流

现在这个本地项目的流程可以概括成下面 7 步：

### Step 1. 准备数据目录
数据放在：
- `data/stock_data/`

可见文件包括：
- `pdf_reports/`
- `questions.json`
- `subset.csv`
- `debug_data/`
- `databases/`
- `answers_*.json`

说明这里已经不是比赛官方目录，而是一个面向中文企业材料的新数据集。

---

### Step 2. PDF 解析
可有两条路：

#### 路线 A：Docling 解析
通过 `src/pdf_parsing.py` 结构化解析 PDF，生成 JSON。

#### 路线 B：MinerU 解析
通过 `src/pdf_mineru.py` 生成 Markdown，再放到统一目录中。

#### 在 `pipeline.py` 中的体现
`Pipeline.export_reports_to_markdown()`：
- 调 MinerU
- 找到 `full.md`
- 移动到 `debug_data/03_reports_markdown`

---

### Step 3. 报告规整
使用：
- `src/parsed_reports_merging.py`

把结构化块进一步清洗成页文本。

这里的目标不是保持原始 PDF 结构，而是得到适合检索的文本页面。

---

### Step 4. 文本切分
使用：
- `src/text_splitter.py`

把每页 Markdown/文本切成 chunk，输出到：
- `databases/chunked_reports/`

这样就可以后续建向量库和 BM25 索引。

---

### Step 5. 建库
使用：
- `src/ingestion.py`

生成：
- FAISS 向量库
- BM25 索引

本地版里主要是向量库优先，BM25 作为可选补充。

---

### Step 6. 问题处理
使用：
- `src/questions_processing.py`
- `src/retrieval.py`
- `src/reranking.py`
- `src/api_requests.py`

先识别问题里的公司名，再按公司限定检索，再构建 RAG 上下文，再交给 Qwen 生成答案。

---

### Step 7. 生成提交文件 / 调试文件
`QuestionsProcessor` 会输出：
- `answers*.json`
- `answers*_debug.json`

并保存：
- 最终答案
- 引用页码
- 推理过程
- 调试信息

---

## 四、关键模块的改写说明

### 1. `src/pipeline.py`

这是本地版最重要的调度层。

#### 新增/调整点
- 路径统一改为 `stock_data`
- 增加 `export_reports_to_markdown()`，接 MinerU 输出
- 增加 `RunConfig` 中的国内模型配置
- 默认答题模型改为 `qwen-turbo-latest`
- `api_provider` 默认改成 `dashscope`
- 新增 `max_config`，用于“检索 + 父文档 + rerank + Qwen”组合

#### 说明
冠军项目原来更像是一个多配置实验平台；
本地版在此基础上，已经把“生产可跑”重点放到中文企业知识库上。

---

### 2. `src/api_requests.py`

这是模型 API 的统一适配层。

#### 改写要点
- 新增 `BaseDashscopeProcessor`
- `APIProcessor` 支持 dashscope
- `get_answer_from_rag_context()` 能兼容国内模型返回格式
- 支持结构化输出的兜底解析

#### 影响
这让上层 `QuestionsProcessor` 不需要关心底层到底是 OpenAI、Gemini 还是 Qwen。

---

### 3. `src/retrieval.py`

这是检索层的核心。

#### 改写要点
- `VectorRetriever` 支持 `embedding_provider="dashscope"`
- 使用 DashScope 的 `text-embedding-v1`
- 文档定位方式基于 `company_name` + `sha1`
- `retrieve_all()` 可直接返回公司全文页

#### 额外变化
本地版检索逻辑明显强化了“**公司限定检索**”这个前提：
- 先识别公司
- 再只在该公司文档中做向量召回

这非常适合企业年报问答。

---

### 4. `src/reranking.py`

#### 改写要点
- `LLMReranker` 默认 provider 是 `dashscope`
- 使用 Qwen 进行重排
- 支持单块和多块重排
- 最终按向量分数 + LLM 分数加权融合

#### 说明
这一步很关键，因为中文财报问题通常“相关片段很多，但真正答案很局部”。
重排可以显著提升最终上下文质量。

---

### 5. `src/questions_processing.py`

#### 改写要点
- 支持从 `subset.csv` 中抽取中文公司名
- 支持多公司比较问答
- 支持新挑战 pipeline 格式
- 支持页码引用校验
- 支持答案详情与调试输出

#### 关键逻辑
1. 从问题文本中匹配公司名
2. 单公司问题：直接检索并作答
3. 多公司问题：先拆成多子问题，再分别回答，最后综合比较
4. 对 LLM 产生的页码引用进行校验，减少幻觉引用

#### 中文场景适配
公司名不再是英文引号里的 name，而是直接在中文问题中匹配 `subset.csv` 中的中文公司名。

## 五、本地版相对冠军项目最值得注意的“中文化”细节

### 1）公司名抽取改成中文直接匹配
原冠军项目更偏“英文/引号式公司名抽取”。
本地版通过：
- `self.companies_df['company_name']`
- 在问题文本里直接做包含匹配

这更适合中文问句，例如：
- “中芯国际的营收是多少？”
- “比亚迪和宁德时代哪家毛利率更高？”

---

### 2）页码引用和 `subset.csv` 绑定
回答后会把页码转成参考文献格式：
- `pdf_sha1`
- `page_index`

这样提交格式和可追踪性都更好。

---

### 3）表格、页眉页脚、脚注更偏中文报告处理
`parsed_reports_merging.py` 对：
- table
- list_item
- footnote
- page_header
- section_header
做了较强的规整。

这有助于把中文 PDF 还原成更平滑的语义文本。

## 六、从“冠军项目”到“本地改写版”的差异总结

可以把差异压缩成一句话：

> 原冠军项目是一个面向比赛的数据检索问答系统；本地版则是一个面向中文企业财报/研报的国产模型 RAG 方案。

更具体地说：

| 维度 | 冠军项目 | 本地改写版 |
|---|---|---|
| PDF 解析 | Docling 为主 | Docling + MinerU |
| 数据语言 | 英文年报/竞赛数据 | 中文财报、研报 |
| Embedding | OpenAI 为主 | DashScope embedding |
| 问答模型 | OpenAI / Gemini / IBM | Qwen / DashScope |
| 重排 | OpenAI / Jina / LLM | Qwen 重排 |
| 公司匹配 | 更偏比赛原始字段 | 中文公司名直接匹配 |
| 输出 | 比赛提交格式 | 提交格式 + 调试文件 + 本地分析 |

- PDF/Markdown 解析
- 报告规整
